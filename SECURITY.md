## 📋 Сообщение об уязвимости

Если вы обнаружили уязвимость безопасности, пожалуйста, следуйте этим рекомендациям:

### ❌ НЕ ДЕЛАЙТЕ

- **Не создавайте публичный Issue** с описанием уязвимости
- **Не публикуйте информацию** об уязвимости в социальных сетях
- **Не отправляйте информацию** в публичные чаты

### ✅ ДЕЛАЙТЕ

- **Отправьте email** на [security@quantum-horizon.dev](mailto:security@quantum-horizon.dev) (если настроено)
- **Создайте приватный Advisory** в [GitHub Security](https://github.com/QuadDarv1ne/quantum-horizon/security/advisories/new)
- **Используйте форму** [Private vulnerability reporting](https://github.com/QuadDarv1ne/quantum-horizon/issues/new/choose)

---

## 📝 Что включать в отчёт

Для максимально быстрого решения проблемы, пожалуйста, включите:

1. **Описание уязвимости** — подробное описание проблемы
2. **Шаги воспроизведения** — как можно воспроизвести уязвимость
3. **Версия проекта** — какая версия затронута
4. **Влияние** — какой потенциальный ущерб может быть нанесён
5. **Доказательство концепции** (если возможно) — код или скриншоты
6. **Рекомендации** — ваши предложения по исправлению (опционально)

## 🛡️ Типы уязвимостей

### Критические
- Удалённое выполнение кода (RCE)
- SQL-инъекции
- XSS (межсайтовый скриптинг)
- CSRF (межсайтовая подделка запросов)
- Аутентификация/авторизация

### Средние
- Раскрытие информации
- Недостатки валидации данных
- Проблемы с сессиями

### Низкие
- Best practice нарушения
- Информационные утечки без критического влияния

## 🔐 Безопасность в проекте

### Используемые меры безопасности

- **NextAuth.js** — современная аутентификация
- **Prisma ORM** — защита от SQL-инъекций
- **TypeScript** — статическая типизация для предотвращения ошибок
- **CSP headers** — Content Security Policy
- **HTTPS** — шифрование трафика (в production)

### Зависимости

Мы регулярно обновляем зависимости и используем:

```bash
npm audit
npm audit fix
```

Автоматические обновления через:
- **Dependabot** — автоматические PR для обновлений безопасности
- **Renovate** — мониторинг устаревших пакетов

## 📜 Политика раскрытия информации

Мы следуем принципу **ответственного раскрытия**:

1. Получение отчёта об уязвимости
2. Подтверждение и анализ
3. Разработка исправления
4. Выпуск патча
5. Публичное объявление (после выпуска)
